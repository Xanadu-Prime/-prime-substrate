1. Purpose
The substrate lifecycle defines the existence cycle of substrate artifacts.
It governs how structural entities (directories, files, invariants, axioms, governance rules) move through time inside the substrate.

This lifecycle ensures that substrate evolution is deterministic, reversible, and operator‑controlled.

2. Lifecycle Overview
All substrate artifacts pass through four canonical phases:

Emergence — the artifact is declared

Stabilization — contradictions are resolved

Persistence — the artifact becomes part of the substrate’s stable state

Dissolution — the artifact is retired, archived, or removed

These phases define the substrate’s temporal geometry.

Open lifecycle rules

3. Phase Definitions
3.1 Emergence
An artifact emerges when:

the operator explicitly declares it

its structural role is defined

its placement is known

its lineage origin is established

Emergence is the substrate’s creation event.

To expand emergence:
Define instantiation phase

3.2 Stabilization
An artifact stabilizes when:

contradictions are surfaced

the Razor resolves them

naming and structure are fixed

lineage is validated

Stabilization is the substrate’s contradiction‑free state.

To deepen Razor mechanics:
Open razor principles

3.3 Persistence
An artifact persists when:

it has a stable identity

it participates in substrate inheritance

it is referenced by governance or doctrine

it has no unresolved contradictions

Persistence is the substrate’s long‑term memory.

To expand persistence semantics:
Explain lifecycle semantics

3.4 Dissolution
An artifact dissolves when:

it is no longer required

its lineage can be cleanly reversed

no active dependents exist

the operator explicitly authorizes dissolution

Dissolution is not deletion —
it is a controlled end‑of‑life event.

To deepen dissolution:
Define resolution phase

4. Lifecycle Invariants
The substrate enforces the following invariants:

no artifact may skip stabilization

no artifact may persist with contradictions

no artifact may dissolve implicitly

lineage must remain continuous

operator intent governs all transitions

These invariants maintain substrate integrity across time.

5. Transition Rules
Valid transitions:

Emergence → Stabilization

Stabilization → Persistence

Persistence → Dissolution

Invalid transitions:

Emergence → Persistence

Stabilization → Dissolution

Dissolution → Persistence

Invalid transitions must be rejected or escalated through DSR‑1.

To review DSR‑1:
Open DSR‑1

6. Artifact Classes
The lifecycle applies to:

structural entities (directories, files)

governance rules

axioms

lifecycle definitions

doctrine stubs

Prime‑era historical artifacts

Each class inherits the same lifecycle, but with different stabilization requirements.

7. Out of Scope
The substrate lifecycle does not define:

runtime behavior

operator workflow

scanner/vessel execution

TUI interactions

Those belong to higher layers.
