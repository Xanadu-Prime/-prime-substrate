Technical Overview — Xanadu Prime (v1.0)
1. Purpose
This document provides a technical, system‑level overview of Xanadu Prime — the first operational instance of the Xanadu architecture.
It describes the substrate, governance, addressing, transclusion, lineage, and workflow mechanics that make Prime a functioning hypertext system.

This is the engineering‑grade introduction to Prime.

5. Addressing Model
Prime uses stable, permanent addresses for all artifacts:

documents

versions

spans

transclusions

links

Addresses never change.
They are location‑independent, time‑independent, and context‑independent.

This enables:

perfect referential integrity

stable transclusion

immutable lineage

deterministic retrieval

6. Versioning Model
Prime implements deep versioning:

nothing is overwritten

every change creates a new version

versions form a lineage chain

all versions remain addressable

transclusions can target any version

This ensures historical fidelity and non‑destructive evolution.

7. Transclusion Model
Prime implements true transclusion, defined as:

reference‑based reuse

no copying

automatic propagation of updates

preserved provenance

stable bidirectional links

Transclusion is the core mechanism that differentiates Prime from copy‑based systems like the web.

8. Link Model
Prime links are:

bidirectional

lineage‑aware

non‑destructive

structurally governed

Every link knows:

where it originates

where it points

which version it references

how it participates in lineage

Prime defines a deterministic workflow:

11. Interoperability
Prime can interoperate with external systems through adapters, which translate:

copy‑based models → reference‑based models

mutable systems → immutable lineage

URL‑based addressing → stable addressing

Interoperability is possible but requires structural translation, not direct equivalence.
